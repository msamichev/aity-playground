# Retrospectives

Командные ретроспективы **после каждого релиза** (тег `vX.Y.Z`). Не реже раза в 2 недели даже если релизов меньше.

**В team+ai личные посессионные ретро не коммитятся в командный репо** — для них есть локальный scratch (например, `~/notes/<project>/`) или приватная папка в `.gitignore`. Иначе через 3 месяца `retrospectives/` становится нечитаемой свалкой посессионных заметок 4-7 разработчиков.

## Зачем

- **Команда видит тренд** через DORA Four Keys (Deployment Frequency, Lead Time, Change Failure Rate, MTTR). Через 3-5 ретро становится очевидно, амплифицирует ли AI её сильные стороны или дисфункцию (DORA 2025: «AI is an amplifier»).
- **Передача опыта между релизами.** Ни AI в новом чате, ни команда через месяц не помнят, что было.
- **Заземление action items** — что меняем в процессе с владельцами и сроками.

## Имя файла

`retrospectives/YYYY-MM-DD-release-vX.Y.Z.md` — один файл на релиз.

Где:
- `YYYY-MM-DD` — дата ретро-митинга (не обязательно дата релиза).
- `vX.Y.Z` — версия из последнего тега или `VERSION` файла.

Пример: `2026-05-22-release-v0.5.0.md`.

## Формат

6 секций:

1. **Что вошло в релиз** — список фич/фиксов с `Refs: PROJ-XXX`.
2. **Что прошло хорошо** — практики и решения, которые сработали.
3. **Что можно было лучше** — конкретные ситуации.
4. **Что меняем в процессе** — действия с владельцами и сроками.
5. **DORA Four Keys** — таблица 4 ключей + контекст изменений. **Обязательно.**
6. **Follow-up** — action items, переезжающие в backlog/планы.

Полный шаблон — [`TEMPLATE.md`](TEMPLATE.md).

## Создание

Skill `/retro` создаёт файл с правильным именем и шаблоном. Заполняется командно (например, на retro-митинге после релиза).

Helper для DORA-данных из git: `bash scripts/dora-collect.sh` — выводит сырые числа (Deployment Frequency, медиану Lead Time), которые остаётся проверить и вписать. CFR и MTTR — вручную из incident tracker.
