# Методология работы команды 4-7 человек + AI

> Каркас для проектов, где небольшая команда работает совместно, и **каждый разработчик активно использует AI-ассистента** (Claude Code, Cursor, Aider, Codex CLI) как соавтора. Командная работа организована по GitLab Flow с обязательными Merge Request и защищённым `main`.
> Не зависит от языка программирования. Подходит для любого типа продукта.
> Версия — см. [VERSION](VERSION). История изменений — [CHANGELOG.md](CHANGELOG.md). Миграции между версиями — [migrations/](migrations/).

**Краткая характеристика:** это **Spec-Driven Development для команды с AI** — спецификация (идея, принципы, ADR) первична, код выводится из неё, разработчики используют AI как соавтора, координация через MR с обязательным человеческим ревью. Методология базируется на [`solo+ai`](https://github.com/msamichev/methodology-ai/tree/main/solo+ai) и адаптирована под GitLab Flow и параллельную работу нескольких разработчиков (раздел [«Командная специфика»](#12-командная-специфика)).

В мейнстриме SDD стал термином в 2025-2026: GitHub Spec Kit, Anthropic Skills, AGENTS.md как открытый стандарт — все они продвигают одну идею: AI работает гораздо лучше, когда сначала есть **точная и доступная спецификация**, а не «попробуй угадать». Эта методология — практическое воплощение SDD для команды разработчиков с AI.

## TL;DR

Эта папка — **готовая заготовка для нового командного проекта**. Распаковываешь в пустую папку, **сваливаешь в `inbox/` всё, что у тебя есть про идею** (чаты с AI, заметки, документы, тех. задание), и **первый разработчик** говорит Claude `/init-project`. Он прочитает inbox, разложит мысли по полочкам, создаст `CODEOWNERS` и MR-шаблон. Остальные разработчики после клонирования запускают `/onboard-developer` — настроят свои локальные конфиги, не затронув командные.

Дальше — обычная работа в feature-ветках с MR:
- `/adopt-stack` — когда определились со стеком, создаёт структуру кода и настраивает проверки качества (делает первый разработчик; команда подтверждает через MR).
- `/plan <название>` — план новой фичи (по одной задаче из внешнего трекера). Спрашивает ID задачи.
- `/save-all` — коммит и push **в свою feature-ветку** (не в `main`).
- `/open-mr` — открыть Merge Request в `main` через `glab` или печатает URL для GitLab UI.
- `/adr <название>` — записать архитектурное решение. Индекс ADR обновляется автоматически.
- `/retro` — общекомандная ретроспектива после релиза.

Всё устроено так, чтобы:
- **Контекст не терялся между сессиями и между разработчиками.** Любой AI читает `AGENTS.md`, для личных предпочтений — `CLAUDE.local.md` (gitignored).
- **Решения были задокументированы.** Через ADR — короткие записи «почему сделали так».
- **Документы были связаны.** В каждом — раздел `## Связки`, связи проверяются автоматически.
- **Параллельная работа не порождала merge-конфликтов на документации.** Индекс ADR и CHANGELOG-фрагменты — по одному файлу на запись (паттерн GitLab).
- **Защищённый `main` страховал команду.** Push в `main` блокируется локально (`pre-push-guard.sh`) и на GitLab (branch protection). Любые изменения — через MR + approve другого разработчика + зелёный pipeline + squash on merge.
- **Качество держалось автоматикой, а не дисциплиной.** Pre-commit и CI ловят то, что глаз не поймает; AI-ревью (опционально, отдельный шаблон) — первая линия защиты от шума.

## 1. Как начать новый проект

### Шаг 1. Распакуй каркас в новую папку

```bash
# 1. Распакуй архив (любой из двух — что удобнее)
tar -xzf methodology-seed.tar.gz   # для Linux/macOS
unzip methodology-seed.zip         # для Windows или если нет tar

# 2. Скопируй содержимое в новую папку проекта.
#    Точка в конце 'methodology-seed/.' означает «всё содержимое, включая скрытые файлы
#    (.gitignore, .pre-commit-config.yaml, .claude/) — но не саму папку».
cp -r methodology-seed/. my-new-project/

# 3. Удали исходник распаковки (он больше не нужен)
rm -rf methodology-seed/

# 4. Перейди в новый проект и инициализируй git
cd my-new-project
git init
```

> **Совет:** если у тебя ещё нет папки `my-new-project/`, самый короткий вариант — просто переименовать:
> ```bash
> mv methodology-seed my-new-project
> cd my-new-project
> git init
> ```
> Команды с `cp` нужны, если папка `my-new-project/` уже существует (например, ты создал её заранее или клонировал пустой репозиторий из GitHub/GitLab).

### Шаг 2. Положи в `inbox/` всё, что у тебя есть про идею

**Это самое важное и самое часто пропускаемое.** Перед тем как запускать что-либо, собери в одну папку:

- Экспорты чатов с любыми AI, где обсуждалась идея (ChatGPT, Claude, Gemini)
- Голосовые расшифровки (если надиктовывал мысли)
- Заметки в txt/md
- PDF, презентации, скриншоты
- Конкурентный анализ
- Любые «разрозненные мысли»

**Не сортируй ничего.** Просто свали всё в `inbox/`. AI прочитает и разложит сам.

Если есть приватное (личные данные, секреты) — положи в `inbox/private/`, эта подпапка в `.gitignore` и не попадёт в репозиторий.

### Шаг 3. Открой проект в Claude Code и скажи `/init-project`

Claude:
- Прочитает `inbox/`
- Заполнит 5 разделов в `docs/idea/` (idea, research, principles, mvp, risks)
- Создаст 3 продуктовых документа в `docs/product/` (vision / audience / goals)
- Создаст первый ADR и первую ретроспективу
- Где данных не хватит — задаст тебе короткие уточняющие вопросы

После этого у тебя — **полноценная Phase 0** (всё описано, кода нет).

### Шаг 4. Когда идея проработана — `/adopt-architecture`

Когда `docs/idea/` наполнена и тебе понятны главные развилки (но не конкретные технологии) — запусти `/adopt-architecture`. Claude:
- Задаст серию архитектурных вопросов (тип системы, стиль взаимодействия, контракт API, БД, авторизация, деплой, наблюдаемость, NFR)
- На каждое значимое решение создаст отдельный ADR в `docs/adr/`
- Обновит `docs/architecture/overview.md` с C4-диаграммой на Mermaid
- Заполнит подпапки `docs/architecture/core/`, `data/`, `flows/`, `integrations/`, `nfr/`, `roadmap/`
- В конце запустит `/skills-suggest` — рекомендации skills под выбранную архитектуру

Кода ещё нет. Только архитектурные паттерны.

### Шаг 5. Когда понятен стек — `/adopt-stack`

Когда после архитектурной фазы ты решил, на чём конкретно пишешь (например «бэк на Kotlin/Spring, фронт на React, БД Postgres»), запусти `/adopt-stack`. Claude:
- Создаст папки под код (зависит от типа проекта)
- Настроит pre-commit и CI под конкретные языки
- Заполнит секцию «Стек» в `AGENTS.md`
- Запишет ADR с обоснованием выбора
- Дополнит `.claude/expected-tools.md` стек-специфичными инструментами
- В конце запустит `/doctor` и `/skills-suggest`

После этого можно писать код.

### Шаг 6. Обычная работа

- Новая фича → `/plan <название>` (в начале запускает `/doctor` под `phase-2-feature` — пред-фичный чек окружения) → код → `/full-ahead` (всё прогнать и запушить)
- Архитектурное решение → `/adr <название>`
- Конец сессии → `/retro`
- Появились новые материалы в `inbox/` → `/sync-idea` (подтянет в `docs/idea/`)
- Сомнения в плане → `/critic` (раскритикует с 4 разных позиций)
- Раз в месяц — `/skills-audit` (что из установленных skills реально использовал)
- Окружение проверить — `/doctor` (что установлено, что не хватает)

Всё. Подробности — ниже.

## 2. Пять принципов

### 1. Идея важнее кода

Сначала разбираемся **что и для кого строим**. Только потом — на каком языке. Если идея непонятна — никакой стек не поможет.

### 2. Документы связаны между собой

Каждый осмысленный документ обязан иметь раздел `## Связки` в конце — со ссылками на смежные. Это превращает разрозненную папку в **граф знаний**: можно прыгать по связям и видеть, что на что влияет. Битые ссылки и забытые связки ловит скрипт `scripts/validate-links.sh`.

### 3. Снимок против живого

- `docs/idea/` — **снимок мысли на момент инициализации**. Замёрз. Изменения — только через ADR, не правкой снимка.
- `docs/architecture/` — **живой документ**, растёт по мере прояснения архитектуры.

Они не пересекаются: idea — про «что и зачем», architecture — про «как технически».

### 4. Память AI ≠ память проекта

Все факты про проект (стек, решения, продукт) — **в файлах**. Никогда не полагайся на то, что AI «помнит» из прошлых разговоров: каждый чат у него начинается с чистого листа, он перечитывает `AGENTS.md` и документы.

В личной памяти AI (которая у Claude Code есть) — **только привычки и стиль работы**: «предпочитаю русский в чате», «терминал на порту 5173». Не факты проекта.

### 5. Качество — автоматикой, а не дисциплиной

Команда из 4-7 разработчиков быстро устаёт от ручных проверок, особенно когда гейт перед каждым MR — 30 минут. Поэтому всё критичное разнесено по уровням (см. [ADR-1432](https://github.com/msamichev/methodology-ai/blob/main/docs/adr/20260518-1432-split-local-ci-into-tiers.md) + [team+ai-специфика](https://github.com/msamichev/methodology-ai/blob/main/docs/adr/20260522-1400-team-ai-ci-tiers-with-gitlab-nightly-deep.md)):

- **Fast** ([`scripts/ci-fast.sh`](scripts/ci-fast.sh)) — секунды, каждый commit через git hook: формат, линт, секреты, формат коммитов, валидация связок.
- **Push** ([`scripts/ci-push.sh`](scripts/ci-push.sh)) — минуты, перед push в feature-ветку: билд + проверка типов, SAST + gitleaks, SCA + license (trivy), тесты + coverage. Дефолт `/full-ahead`.
- **Deep** ([`scripts/ci-deep.sh`](scripts/ci-deep.sh)) — десятки минут. **Основное место запуска в команде — GitLab CI nightly job** `security-deep:nightly` (см. [`.gitlab-ci.yml`](.gitlab-ci.yml) + операционная инструкция как настроить schedule). Локально — opt-in через `scripts/local-ci.sh --deep` для отладки конкретной регрессии.

Плюс короткий [`playbooks/self-review.md`](playbooks/self-review.md) — 7 смысловых пунктов, читает автор **перед `/open-mr`** (не перед каждым checkpoint push). Дополняет [`merge-coordinator`](.claude/agents/merge-coordinator.md) subagent: тот делает структурные проверки (rebase, ADR, changelog-фрагмент, trailers), self-review — смысловые.

Если автоматика не поймает — будет в коде. Не надейся на собственную внимательность.

## 3. Четыре фазы жизни проекта

```
Phase 0: Discovery   →   /adopt-architecture   →   Phase 1: Stack   →   обычная работа   →   Phase 2: Steady state
   (нет кода)               (Phase 0.5:                (структура есть)
                             архитектурные
                             паттерны → ADR)
```

**Phase 0: Discovery.** После `/init-project` есть всё, кроме кода. `AGENTS.md` помечен «стек ещё не выбран». В этой фазе наполняется идея, обсуждаются варианты, пишутся первые ADR. Длиться может от часа до недель.

**Phase 0.5: Architecture Adoption.** Когда идея проработана, перед выбором конкретных технологий — запускается `/adopt-architecture`. Архитектурные развилки (монолит/микросервисы, request-response/event-driven, REST/GraphQL, реляционная/документная БД, и т.д.) фиксируются как отдельные ADR. Обновляется `docs/architecture/overview.md` с C4-диаграммой. Конкретного стека ещё нет — только архитектурные классы.

**Phase 1: Stack Adoption.** Когда понятно, на чём пишем под выбранную архитектуру — `/adopt-stack` создаёт структуру под тип проекта и настраивает проверки. После этого можно писать код.

**Phase 2: Steady state.** Стандартный цикл: план → код → тесты → коммит → ретро. Решения через ADR. Стек поменялся — новый ADR.

**Почему архитектура отдельной фазой:** в реальной практике сначала ты решаешь архитектурные паттерны, и только потом — конкретные технологии под них. Если объединить — конкретика стека размывает архитектурное мышление, и важные развилки проходят мимо.

## 4. Что лежит где

Полная карта проекта — в [`AGENTS.md §2`](AGENTS.md#2-карта-проекта) (это единственный источник правды для структуры, чтобы не плодить дубли). Краткие пояснения логики папок:

- **`docs/product/`** — 3 коротких файла (vision / audience / goals). Что строим и для кого, на 1 абзац.
- **`docs/idea/`** — 5 разделов идейного ядра (frozen snapshot). Не редактируется спонтанно; изменения — через ADR.
- **`docs/architecture/`** — живая архитектура. Растёт по мере прояснения. C4-диаграммы + подпапки (core/data/flows/integrations/nfr/roadmap/stack).
- **`docs/adr/`** — все архитектурные и техно-значимые решения.
- **`plans/`** — один план = одна фича. Шаблон в [`plans/TEMPLATE.md`](plans/TEMPLATE.md).
- **`retrospectives/`** — после каждой сессии. Короткие 5-пунктовые ретро.
- **`playbooks/`** — универсальные процедуры команд (читаются любым AI).
- **`.claude/`** — тонкие skill-обёртки для Claude Code + сгенерированные настройки.
- **`scripts/`** — `validate-links`, `rename-doc`, `local-ci`, `sync-claude-md`, `gen-claude-deny`, `pre-push-guard`.

Папки с кодом (`backend/`, `frontend/`, `src/` и т.д.) появляются на `/adopt-stack`.

## 5. Как фиксируем решения: ADR

**ADR** (Architecture Decision Record) — короткий документ «почему мы решили сделать так». Файл в `docs/adr/`.

**Когда писать:**
- Выбор языка, базы данных, фреймворка
- Принцип работы с авторизацией, кэшем, миграциями
- Любой выбор, на который через 6 месяцев кто-то спросит «почему так?»

**Когда не писать:**
- Имена переменных, мелкие детали кода
- Идейные продуктовые решения, не имеющие технических последствий (это в `docs/idea/01-idea.md` или `03-principles.md`)
- Одноразовые эксперименты

**Имя файла:** `YYYYMMDD-HHmm-короткое-описание.md`, например `20260517-1430-choose-postgres.md`. Дата+время вместо нумерации — чтобы при работе в разных ветках не было коллизий.

**Содержит:** контекст («что решаем»), решение, альтернативы (что рассмотрели и отказались), последствия (плюсы, минусы, что тянет за собой).

Шаблон — [`docs/adr/template.md`](docs/adr/template.md). Удобнее создавать через `/adr`.

**Если решение пересматривается** — старый ADR не редактируем, пишем новый. В старом меняем только статус на `Superseded by <новый-ADR>`. История остаётся.

## 6. AGENTS.md и команды-помощники

### AGENTS.md — контракт для AI

Файл в корне (до 300 строк), который AI читает в каждой новой сессии. В нём — одностраничный обзор проекта: что за продукт, стек, где что лежит, индекс ADR, конвенции (имена файлов, формат коммитов, безопасность), правила работы.

**Главная идея:** ты не пересказываешь AI заново каждый чат, что за проект. AI открывает `AGENTS.md` — и в контексте.

Рядом — `CLAUDE.md`, это симлинк на тот же файл. Так что Claude Code, Codex и Gemini CLI читают одно и то же. Редактируется **только `AGENTS.md`**.

> **Windows caveat:** создание симлинков на Windows требует Admin-прав или включённого Developer Mode (Settings → Update & Security → For developers). Если симлинки отключены в git (`core.symlinks=false`), `CLAUDE.md` создаётся как обычная копия, которая автоматически синхронизируется pre-commit хуком `scripts/sync-claude-md.sh`. На macOS/Linux симлинк работает «из коробки».

### Команды-помощники: универсальные для любого AI

Чтобы команды работали **не только в Claude Code**, но и в Codex, DeepSeek, Minimax, Cursor, Gemini, Aider и других AI — используется **двухуровневая архитектура**:

```
playbooks/<name>.md            ← Источник правды (обычный Markdown)
  ↑                              читается любым AI
  │ ссылается на
  │
.claude/skills/<name>/SKILL.md ← Тонкая обёртка для Claude Code
                                 (YAML + ссылка на playbook)
```

**Зачем так:**
- **Контент один** — в `playbooks/<name>.md`. Не дублируется.
- **Claude Code** получает приятный авто-триггер: можно говорить триггерную фразу естественным языком, Claude сам подбирает skill по семантике `description`.
- **Другие AI** читают `AGENTS.md` § 6 — там таблица «триггерная фраза → playbook». Распознают, открывают, выполняют.
- **Добавить новый AI** (например, Cursor) — генерируешь ему обёртку из playbook'а, по аналогии с `.claude/skills/`. Сам контент не трогаешь.

Самые важные команды:

| Команда | Что делает |
|---|---|
| `/init-project` | Однократно при старте — раскладывает `inbox/` по полочкам |
| `/adopt-architecture` | Перед выбором стека — архитектурные паттерны → ADR |
| `/adopt-stack` | Однократно при выборе стека — создаёт структуру кода |
| `/doctor` | **Фазно-осознанная** проверка окружения: что установлено, что отсутствует, что конфликтует, какие skills рекомендуются под текущую фазу |
| `/skills-suggest` | Фокусированный разбор только по skills (детально под фазу + стек) |
| `/skills-audit` | Раз в месяц — что из установленных skills использовал |
| `/plan <название>` | Создать план фичи |
| `/retro` | Ретроспектива сессии (5 пунктов) |
| `/adr <название>` | Записать архитектурное решение |
| `/save` / `/save-all` | Коммит / коммит + push |
| `/full-ahead` | «Полный вперёд»: всё прогнать → исправить → запушить |
| `/critic` | Раскритиковать текущий план с 4 разных позиций |
| `/sync-idea` | Подтянуть новые материалы из `inbox/` в `docs/idea/` |
| `/check-links` | Подробный отчёт по графу связей |

Полная таблица с триггерными фразами — в [`AGENTS.md §6`](AGENTS.md). Сами playbooks — в [`playbooks/`](playbooks/). Каталог рекомендуемых skills — [`.claude/recommended-skills.md`](.claude/recommended-skills.md). Каталог ожидаемых инструментов — [`.claude/expected-tools.md`](.claude/expected-tools.md).

## 7. Граф связей: правило «Связки»

В каждом осмысленном документе в `docs/architecture/`, `docs/idea/`, `docs/adr/`, `docs/product/` — внизу есть раздел `## Связки` со ссылками на смежные документы.

Это превращает папку с файлами в **граф знаний**, где можно прыгать по связям и видеть, что на что влияет.

**Синтаксис ссылок** — обычный Markdown: `[название](относительный/путь.md)`. Не используем Obsidian-стиль `[[...]]` — обычный Markdown лучше работает с GitHub, IDE и линтерами.

**Скрипт `scripts/validate-links.sh`** (запускается автоматически на pre-commit и в CI):
- Битая ссылка (файл не существует) → **ошибка**
- Файл в смысловой папке без раздела `## Связки` → **ошибка**
- Файл, на который никто не ссылается → **предупреждение** (бывает нормально, например для шаблонов)

Если переименовал документ — `scripts/rename-doc.sh старый/путь.md новый/путь.md` сам обновит все входящие ссылки во всём репозитории.

## 8. Память AI и память проекта

| Где хранится | Что туда идёт |
|---|---|
| **В файлах** (`docs/`, ADR, `AGENTS.md`) | Факты про проект: стек, архитектурные решения, продукт. **Единственный источник истины.** |
| **В личной памяти AI** (Claude Code auto-memory) | Привычки и операционные мелочи: язык чата, порты на машине, личные предпочтения. |

## 9. Качество кода

Локальный CI разнесён по трём уровням — Fast / Push / Deep — чтобы pre-push гейт перед feature-веткой укладывался в минуты, а тяжёлые проверки (mutation, SBOM, container scan) выполнялись централизованно в GitLab nightly, не умножая время команды на N разработчиков. Обоснование — [ADR-1432](https://github.com/msamichev/methodology-ai/blob/main/docs/adr/20260518-1432-split-local-ci-into-tiers.md) + [team+ai-специфика 20260522-1400](https://github.com/msamichev/methodology-ai/blob/main/docs/adr/20260522-1400-team-ai-ci-tiers-with-gitlab-nightly-deep.md).

| Уровень | Скрипт | Где | Что проверяется |
|---|---|---|---|
| **Fast** | [`scripts/ci-fast.sh`](scripts/ci-fast.sh) | локально, каждый commit (git hook) | Формат, линт, секреты, формат коммита, валидация связок |
| **Push** | [`scripts/ci-push.sh`](scripts/ci-push.sh) | локально, перед push в feature-ветку | Билд + проверка типов; SAST + gitleaks; SCA + license (trivy fs); тесты + coverage |
| **Deep** | [`scripts/ci-deep.sh`](scripts/ci-deep.sh) | **GitLab CI nightly** (`security-deep:nightly` в [`.gitlab-ci.yml`](.gitlab-ci.yml)); локально — opt-in для отладки | **Mutation testing**; SBOM (CycloneDX); container scan (trivy image) + IaC (checkov, hadolint) |

Плюс [`playbooks/self-review.md`](playbooks/self-review.md) — 7-пунктный смысловой чек-лист, читает автор **перед `/open-mr`**. Дополняет [`merge-coordinator`](.claude/agents/merge-coordinator.md) subagent (структурные vs смысловые проверки).

**Mutation testing** — главный инструмент против «фальшивых» тестов от AI. Тестер специально портит код в разных местах и смотрит, ловят ли это твои тесты. Если тесты есть, но они не реагируют на ломку кода — они бесполезны. Порог: **75% базовый, 85% для критичных модулей**. Живёт в Deep-уровне (GitLab nightly), чтобы не задерживать каждый MR на десятки минут.

Точные команды под язык подставляются командой `/adopt-stack` в соответствующие скрипты. `scripts/local-ci.sh` остаётся фасадом: без флагов = `ci-push.sh`, `--fast` = `ci-fast.sh`, `--deep` = `ci-push.sh` + `ci-deep.sh` (для локальной отладки).

Команда `/full-ahead` запускает `ci-push.sh`, исправляет автоматические ошибки сама, и пушит в текущую feature-ветку, когда всё зелёное. Перед `/open-mr` автор читает self-review.

## 10. Совместимость с другими AI и IDE

Методология **по умолчанию заточена под Claude Code**, но устроена так, чтобы работать с другими AI и IDE без переписывания контента — только через добавление тонких обёрток.

### Что Claude-специфично, а что универсально

| Компонент | Только Claude Code | Универсально |
|---|---|---|
| `.claude/skills/*/SKILL.md` (формат с YAML и авто-триггером) | ✓ | |
| `.claude/settings.json` (формат deny-list) | ✓ (но генерируется из универсального источника, см. ниже) | |
| `@-imports` в `CLAUDE.md` (синтаксис `@path/to/file.md` для инлайн-подгрузки) | ✓ (опциональное расширение при росте AGENTS.md; в команде применять только если **все** разработчики на Claude Code — см. [playbooks/README.md](playbooks/README.md) раздел «Claude Code @-imports») | |
| `security/dangerous-commands.md` (правила безопасности) | | ✓ (любой AI читает) |
| `CLAUDE.md` | (это симлинк на AGENTS.md) | ✓ |
| auto-memory `~/.claude/projects/...` | ✓ | |
| `AGENTS.md` | | ✓ (open standard, читают многие) |
| `playbooks/*.md` | | ✓ (обычный Markdown) |
| `METHODOLOGY.md`, `README.md`, `docs/` | | ✓ |
| `scripts/*.sh` (validate-links, rename-doc, local-ci, sync-claude-md, gen-claude-deny, pre-push-guard) | | ✓ (bash) |
| `.pre-commit-config.yaml` | | ✓ |
| `.gitignore` | | ✓ |

**Главное:** вся содержательная часть методологии универсальна. Claude-only — это удобства (авто-триггер skills, формат settings.json, привязка через CLAUDE.md). Контент команд (`playbooks/`) и правила безопасности (`security/dangerous-commands.md`) одинаково читаются любым AI.

### Как устроены правила безопасности (универсально + Claude-friendly)

`security/dangerous-commands.md` — единый источник правды:
- **Любой AI** (Codex, Cursor, DeepSeek, Minimax, и т.п.) читает этот файл по ссылке из `AGENTS.md §4.4` и соблюдает правила добровольно.
- **Claude Code** дополнительно получает машиночитаемый `.claude/settings.json`, в который правила запеканы как `permissions.deny` и `permissions.ask`. Это даёт **жёсткую блокировку**, а не «доброжелательное соблюдение».
- **Генератор** `scripts/gen-claude-deny.sh` пересобирает `.claude/settings.json` из источника автоматически на каждом pre-commit. CI проверяет синхронизацию.
- **Git-уровневые защиты** (`scripts/pre-push-guard.sh`) работают независимо от любого AI: блокируют force-push в `main`, грязный рабочий каталог, расхождение `.claude/settings.json` с источником.

Это значит: меняешь правила безопасности — редактируешь только `security/dangerous-commands.md`. Остальное синхронизируется само.

### Поддержка через стандарт AGENTS.md

Эти инструменты понимают `AGENTS.md` как часть открытого стандарта (Agentic AI Foundation / Linux Foundation, 60k+ проектов):

- **Codex CLI** (от OpenAI) — читает `AGENTS.md` автоматически.
- **Gemini CLI** (от Google) — то же.
- **GitHub Copilot** (с августа 2025) — то же.
- **Continue.dev** — нужен один системный промпт со ссылкой на AGENTS.md (см. рецепт в [playbooks/README.md](playbooks/README.md)).
- **Aider** — **требует однократной настройки** `.aider.conf.yml` в корне проекта, чтобы добавить `AGENTS.md` в `read:`. Из коробки aider не подтягивает AGENTS.md автоматически. Рецепт — в [playbooks/README.md](playbooks/README.md).

После настройки (где она нужна) — никакой адаптации контента не требуется: говоришь триггерную фразу из таблицы AGENTS.md §6, AI открывает соответствующий playbook.

### Cursor — нужны обёртки в `.cursor/rules/`

Cursor использует свой формат — `.cursor/rules/<name>.mdc`. Логика та же, что у `.claude/skills/`: тонкая обёртка с указанием на playbook.

**Что нужно сделать однократно (примерно 12 файлов):**

```bash
mkdir -p .cursor/rules
```

Для каждой команды создать файл `.cursor/rules/<name>.mdc` примерно так:

```markdown
---
description: <скопировать description из .claude/skills/<name>/SKILL.md>
alwaysApply: false
---

См. процедуру в [playbooks/<name>.md](../../playbooks/<name>.md). Выполни её пошагово.
```

После этого Cursor подхватывает команды по тому же принципу, что и Claude Code.

(Можно автоматизировать: написать скрипт `scripts/gen-cursor-rules.sh`, который генерирует все 12 файлов из `.claude/skills/`. Скрипт не входит в seed по умолчанию — добавляется когда нужно. См. [playbooks/README.md](playbooks/README.md) для готовых рецептов.)

### Чат-LLM без файлового доступа (DeepSeek, Minimax, GLM, Qwen и т.п.)

Такие AI не «видят» файловую систему проекта сами. Подход:

1. **В начале сессии** — вставить содержимое `AGENTS.md` в системный промпт или первое сообщение. Сказать: «Это контракт проекта. Когда встретишь триггерную фразу из §6 — открой соответствующий playbook».
2. **Когда LLM распознаёт триггер** — попросит у тебя содержимое playbook'а. Вставляешь файл из `playbooks/<name>.md`.
3. **Альтернатива** — использовать клиент-оркестратор, который умеет работать с любым OpenAI-compatible API и читать файлы проекта сам. Например:
   - **aider** — поддерживает DeepSeek, Qwen и любые OpenAI-compatible endpoints, читает файлы автоматически
   - **continue.dev** — то же самое, плюс есть IDE-интеграция
   - **cline** (бывший Claude Dev) — работает в VS Code, поддерживает разные модели

С таким клиентом не-Claude модель ведёт себя как Codex/Gemini — читает AGENTS.md и playbooks сама.

### Что теряется на не-Claude AI

| Фича | Чем заменить |
|---|---|
| Авто-триггер по семантике («давай новое решение» → `/adr` без явной команды) | Нужна точная триггерная фраза из таблицы AGENTS.md §6 или явный `/команда`. |
| `Bypass Permissions` mode | Только Claude Code. У других AI просто нет такой кнопки — действия требуют подтверждения каждый раз. Pre-commit и CI всё равно срабатывают. |
| Auto-memory личных предпочтений | У Cursor есть «memory bank» (`.cursor/memory/`), у aider — `.aider.chat.history.md`. У чат-LLM без CLI — никак, но факты проекта и так в файлах, а привычки можно переносить руками. |
| Deny-list команд с **жёсткой блокировкой** на уровне клиента (`.claude/settings.json`) | У Cursor — settings → Permissions (формат другой, но идея та же). У aider/continue.dev/чат-LLM — встроенной блокировки нет, остаётся «доброжелательное» соблюдение AI правил из `security/dangerous-commands.md` плюс git pre-push hook (`scripts/pre-push-guard.sh`) как страховка. |

### Общий принцип портирования

Источник правды — `playbooks/`. Всё остальное (`.claude/skills/`, потенциальные `.cursor/rules/`, гипотетические `.aider/commands/` и т.п.) — это **обёртки**, ссылающиеся на playbook.

Чтобы добавить поддержку новой IDE:
1. Узнай её формат custom-команд (если есть).
2. Сделай 12 тонких обёрток по аналогии с `.claude/skills/`.
3. Контент команд не трогай — он один.

Конкретные рецепты для Cursor и других — в [playbooks/README.md](playbooks/README.md).

## 10.1 Версионирование методологии

Сама методология версионируется по [Semantic Versioning](https://semver.org/) — отдельно от приложения, которое ты в ней пишешь.

- **[`VERSION`](VERSION)** — одна строка `MAJOR.MINOR.PATCH`. Это **версия методологии** в твоей копии (не версия твоего продукта). Скопировал семь — VERSION фиксирует исходный baseline; затем правишь под себя — bump'ишь и записываешь в CHANGELOG.
- **[`CHANGELOG.md`](CHANGELOG.md)** — история изменений в формате [Keep a Changelog](https://keepachangelog.com/).
- **[`migrations/`](migrations/)** — markdown-гайды для перехода между MINOR/MAJOR версиями. PATCH-ам гайд не нужен.

**Семантика bump'ов** (детальнее — в ADR про версионирование):
- **MAJOR** — breaking: меняется структура папок, обязательные правила, нужны ручные шаги для апгрейда.
- **MINOR** — новые возможности, рефакторинг без breaking, новые playbook'и, обновление дефолтов.
- **PATCH** — фиксы, опечатки, апдейты версий хуков без поломок.

**Как узнать свою версию:**
```bash
cat VERSION                # короткий способ
/doctor                    # первая строка отчёта содержит «Методология: solo+ai vX.Y.Z»
```

При инициализации нового проекта (`/init-project`) текущая версия методологии записывается в bootstrap ADR — это якорь, по которому через полгода можно понять, с какого baseline стартовали и какие migration guides применять.

## 12. Командная специфика

Этот раздел собирает всё, что отличает `team+ai` от `solo+ai`. Если ты работаешь по этой методологии в первый раз — прочитай его целиком, остальная часть документа применима к команде без существенных изменений.

### 12.1 Роли и где они работают

| Роль | Внешний трекер | GitLab |
|---|---|---|
| Аналитик / PM | заводит задачи, формулирует требования и критерии приёмки, ведёт спринт, принимает фичи | читает MR при необходимости (Reporter); владеет `docs/product/`, `docs/idea/` через CODEOWNERS |
| Разработчик | меняет статусы задачи: «В работе» → «На ревью» → «Готово» | создаёт ветку, пишет код в паре с AI, открывает MR, проходит ревью, мержит сам после approve |
| Ревьюер | — | проверяет MR, оставляет комментарии (must fix / suggest / nit), ставит approve |
| QA / тестировщик | меняет статусы: «На тестировании» → «Принято» / возврат, заводит баги как новые задачи | Reporter — смотрит что вошло в `dev`/`staging` |
| Релиз-инженер / тимлид | собирает задачи в «Готово к релизу» | создаёт теги `vX.Y.Z`, запускает деплой `staging` → `prod`, оформляет GitLab Release |
| Maintainer проекта | — | настраивает branch protection, CI/CD, права, CODEOWNERS |

Все роли, для которых это применимо, **используют AI-ассистента** как соавтора. AI работает в основном на стороне разработчика и аналитика (формулировки требований, ревью кода через `/critic`, написание тестов, документация). Ревью кода человеком — обязательно, AI его не заменяет (см. [мета-ADR об отложенном AI-ревью](https://github.com/msamichev/methodology-ai/blob/main/docs/adr/20260521-1100-team-ai-methodology-on-top-of-solo-ai.md)).

### 12.2 Git-модель и MR-флоу

Полные правила — в [`AGENTS.md` §4.3](AGENTS.md#43-git). Кратко:

- `main` защищён, прямой push запрещён локально (`scripts/pre-push-guard.sh`) и на GitLab (branch protection).
- Любое изменение — через feature-ветку `<тип>/<ID-задачи>-<slug>` + Merge Request.
- MR требует: 1 approve от другого разработчика (не автора), все обязательные threads разрешены, pipeline зелёный, squash on merge, удаление исходной ветки.
- Branch lifetime ≤ 1 неделя (ориентир 3-5 дней). Hammant ([trunkbaseddevelopment.com](https://trunkbaseddevelopment.com/)) рекомендует ≤2 дня — это допустимый ужесточённый вариант.
- Релизные теги `vX.Y.Z` защищены: не удаляются, не переписываются.
- Полный регламент работы с GitLab (роли, среды, hotfix, чеклисты) — отдельный документ в команде (например, `docs/gitlab-workflow.md` или wiki). Методология совместима с регламентом v3.0 и не переписывает его.

### 12.3 Что отличается от solo+ai

| Аспект | `solo+ai` | `team+ai` |
|---|---|---|
| Git-модель | push в `main` без MR | feature-ветки + MR + защищённый `main` + squash on merge |
| Branch lifetime | не ограничен (сессия) | ≤ 1 неделя |
| Команды инициализации | `/init-project` (один раз) | `/init-project` (один раз) + `/onboard-developer` (для остальных разработчиков) |
| Открытие MR | — | `/open-mr` |
| `/full-ahead`, `/save-all` | пушат в `main` | пушат в текущую feature-ветку |
| Имена планов / ADR / changelog-фрагментов | timestamp / дата + slug | timestamp / дата + **ID задачи из трекера** + slug |
| Индекс ADR | ручная таблица в `AGENTS.md §3` | `docs/adr/INDEX.md` генерируется скриптом `scripts/build-adr-index.py` |
| CHANGELOG | один аккумулирующий файл | `changelogs/unreleased/*.md` (по одному файлу на MR) → собирается в `CHANGELOG.md` командой `/release` |
| Ретроспективы | после каждой сессии | общекомандная после релиза (`retrospectives/YYYY-MM-DD-release-vX.Y.Z.md`); личные сессии не коммитятся |
| Личные оверрайды AI | — | `CLAUDE.local.md`, `.claude/settings.local.json` (gitignored, создаются `/onboard-developer`) |
| CODEOWNERS | — | обязательно (минимальный шаблон на `AGENTS.md`, `playbooks/`, `scripts/`, `security/`, `docs/adr/`) |
| Аудит экспериментов | — | `/experiments-audit` раз в месяц |
| Autonomous mode | работает в одной репо-сессии | работает **только в собственной feature-ветке** после rebase на `main` |
| AI-ревью в CI | — | отложено в первой версии; шаблон CodeRabbit/Claude-in-GHA планируется отдельным ADR |

Обоснование каждого решения — в мета-ADR [`methodology-ai/docs/adr/20260521-1100-team-ai-methodology-on-top-of-solo-ai.md`](https://github.com/msamichev/methodology-ai/blob/main/docs/adr/20260521-1100-team-ai-methodology-on-top-of-solo-ai.md) (с реальными источниками: DORA 2024, GitClear, Faros, GitLab CHANGELOG conflict crisis, AGENTS.md spec, Anthropic best practices).

## 11. Что НЕ делаем

| Не делаем | Почему |
|---|---|
| Сквозная нумерация ADR (`0001-`, `0002-`) | Коллизии при работе в разных ветках. Используем `YYYYMMDD-HHmm-`. |
| Wikilinks `[[...]]` | Не работают на GitHub, не понимают IDE и линтеры. Стандартный Markdown лучше. |
| Code review через Pull Request для соло | Тяжело и медленно. Pre-commit + CI — единственные гейты. |
| `git add .` без разбора | Только `git add file1 file2` явно. |
