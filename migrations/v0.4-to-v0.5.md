# Миграция team+ai v0.4 → v0.5

Что подтянуть в **уже скопированном проекте** методологии `team+ai`, чтобы получить функциональность v0.5.0 — обязательная секция DORA Four Keys в релизной ретро + helper-скрипт для сырых данных из git + фикс унаследованного шаблона ретро.

## Зачем мигрировать

В v0.5.0 закрыта основная часть [ADR-1700](https://github.com/msamichev/methodology-ai/blob/main/docs/adr/20260522-1700-dora-four-keys-in-team-retro-and-solo-weekplan.md):

1. **DORA Four Keys обязательны в каждой релизной ретро.** Через 3-5 ретро команда видит тренд по четырём ключам, по которым DORA 2025 определяет, амплифицирует ли AI её сильные стороны или дисфункцию.
2. **Helper `scripts/dora-collect.sh`** снимает ручную работу с двух из четырёх ключей (Deployment Frequency + Lead Time из git), оставляя CFR и MTTR на ручной ввод из incident tracker.
3. **Фикс унаследованного бага в шаблоне ретро.** В team+ai 0.1.0 шаблон `retrospectives/TEMPLATE.md` и `README.md` были скопированы из `solo+ai` через `cp -r` и не адаптированы под релизный формат — они описывали посессионную ретро (5 пунктов: Задача / Как решал / Решил? / Что лучше / Как было/стало), что противоречило правилам team+ai из `playbooks/retro.md` («ретро после релиза, файл один на релиз»). Новая v0.5.0 переписывает шаблон под релизный формат.

DORA Four Keys — четыре цифры по [DORA 2025 методологии](https://dora.dev/dora-report-2025/):
- **Deployment Frequency** — как часто релизим в прод.
- **Lead Time** — медиана от первого коммита feature-ветки до production-релиза.
- **Change Failure Rate** — % релизов, потребовавших hotfix/rollback в 48ч.
- **MTTR** — медиана от инцидента до восстановления.

DORA 2025: «AI is an amplifier» — без этих цифр команда не видит, в какую сторону её несёт.

**Без миграции 0.4.x продолжает работать**, но команда не получает DORA-сигнала и продолжает писать ретро в посессионном формате, который не подходит для команды.

## Предусловия

- Проект на team+ai 0.4.0 (security hardening + code-clone).
- Есть git с релизными тегами `vX.Y.Z` (для `dora-collect.sh`). Если тегов нет — скрипт всё равно работает, но Deployment Frequency покажет 0.

## Шаги

1. **Скопируй новый helper-скрипт** из свежего seed:

   ```bash
   cp /path/to/team+ai/scripts/dora-collect.sh scripts/dora-collect.sh
   chmod +x scripts/dora-collect.sh
   ```

2. **Перепиши `retrospectives/TEMPLATE.md`** копией из свежего seed. **Это критично:** старый шаблон описывает посессионную ретро (5 пунктов), новый — релизную (6 секций, включая DORA таблицу). Если у команды уже есть исторические ретро-файлы в старом формате — оставляй как есть (история), новые ретро пиши по новому шаблону.

3. **Перепиши `retrospectives/README.md`** копией из свежего seed (обновлено описание, имя файла `YYYY-MM-DD-release-vX.Y.Z.md`, формат, ссылки).

4. **Обнови `playbooks/retro.md`** из свежего seed. Главное:
   - Шаг 4 пункт «Метрики релиза (опционально)» переименован в **«DORA Four Keys (обязательно)»** и расширен полным шаблоном.
   - В разделе «Правила» добавлен пункт «DORA Four Keys обязательны, даже если приблизительно» с ссылкой на DORA 2025.
   - В Связках добавлены ссылки на `dora-collect.sh`, DORA Report, DORA quickcheck.

5. **Расскажи команде про новый процесс.** Это **главный** шаг — без дисциплины команды DORA не заполняется и пользы нет. Возможные форматы рассказа:
   - Отдельное письмо в Slack/Mattermost с ссылкой на migration guide.
   - Презентация на ближайшей retro-митинге (5 минут): «теперь добавляем 4 цифры, через 3-5 ретро увидим тренд».
   - Закрепить в onboarding (`/onboard-developer`) для новых членов команды.

6. **Обнови `VERSION`** на `0.5.0` руками.

7. **Прогон `scripts/validate-links.sh`** — должен быть зелёным.

8. **Прогон `scripts/dora-collect.sh`** — должен выдать сырые данные за последний период (с последнего тега до HEAD). Если тегов ещё нет — выдаст «0 тегов» без ошибок.

## Как использовать в первый раз

На первой ретро после миграции:

1. Запусти `bash scripts/dora-collect.sh > /tmp/dora-raw.txt` — получи сырые цифры.
2. Открой `retrospectives/YYYY-MM-DD-release-vX.Y.Z.md` (создан через `/retro` skill).
3. В секции «5. DORA Four Keys» заполни таблицу:
   - Deployment Frequency — из вывода скрипта (категория daily/weekly/monthly).
   - Lead Time — медиана из squash-merge коммитов (скрипт показывает список, медиану команда считает сама).
   - Change Failure Rate — вручную из incident tracker (или прокси из скрипта).
   - MTTR — вручную из incident tracker (или N/A если инцидентов не было).
4. Тренд vs прошлой ретро — `→` для первой ретро (базовая точка).
5. В «Контекст изменений» — какой AI-tool использовали, изменилась ли команда, новый продакт.

Через 3-5 ретро тренд будет очевидным.

## Что НЕ нужно делать

- **Не делай DORA опциональной** в новом retro — теряется весь смысл. Если команда не записывает цифры регулярно, тренд не появляется. Если команда сопротивляется — это симптом, обсуди на ретро (см. DORA 2025 раздел про team capabilities).
- **Не вычищай исторические ретро в старом формате** — они часть истории. Только новые пиши по новому шаблону.
- **Не перевешивай ответственность за DORA** на одного человека (release-инженера). Заполнение — команд-ное на ретро-митинге, чтобы все видели тренд.
- **Не пытайся требовать precision** по MTTR без incident tracker. «N/A если не было» — нормальный ответ.

## Откат

1. Восстановить `retrospectives/TEMPLATE.md` и `README.md` из тега `v0.4.0` методологии (или из своего git до миграции).
2. Откатить `playbooks/retro.md` к v0.4.0.
3. Удалить `scripts/dora-collect.sh`.
4. Вернуть `VERSION` на `0.4.0`.
5. Команд-ные ретро, написанные по новому шаблону — оставлять как есть (история). Удалять не надо.

Полностью обратимо.
