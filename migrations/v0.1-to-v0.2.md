# Миграция team+ai v0.1 → v0.2

Кратко: что нужно сделать в **уже скопированном проекте**, чтобы подтянуть изменения версии 0.2.0 методологии `team+ai`.

## Зачем мигрировать

В 0.2.0 добавлен четвёртый read-only subagent `merge-coordinator`, который ловит специфичные для команды ошибки до открытия MR: ветка устарела относительно `origin/main`, ADR с коллидирующим timestamp в чужой ветке, отсутствует changelog-фрагмент для пользовательских изменений, забыт `Refs:` trailer в коммитах. Это закрывает класс проблем, которые `code-reviewer`/`test-runner`/`security-auditor` принципиально не видят (они смотрят на код, не на состояние ветки относительно команды).

Если ты не используешь autonomous mode и `/open-mr` — миграция нестрого обязательна, но рекомендуется как часть нормального обновления методологии.

## Шаги

1. **Скопируй новый subagent** из свежего seed:
   ```bash
   cp /path/to/team+ai/.claude/agents/merge-coordinator.md \
      .claude/agents/merge-coordinator.md
   ```

2. **Обнови два playbook'а** из свежего seed (либо вручную внеси изменения):
   - `playbooks/open-mr.md` — добавлен Шаг 4 «Прогон merge-coordinator subagent» (обязательный), прежний «прогон /critic» переехал в Шаг 4.5.
   - `playbooks/full-ahead.md` — добавлен Шаг 4.4 «`merge-coordinator` subagent», прежний 4.4 «Проверка критерия приёмки» переехал в 4.5.

3. **Обнови `AGENTS.md` §6.1** — в первом маркированном пункте замени `three-gate review через три read-only subagent-а` на `four-gate review через четыре read-only subagent-а`, добавь `merge-coordinator` в перечисление, добавь упоминание `/open-mr` как обязательного gate.

4. **Обнови `VERSION`** в корне проекта:
   ```bash
   echo "0.2.0" > VERSION
   ```

5. **Прогон smoke-теста** (в существующем проекте):
   ```bash
   scripts/validate-links.sh --report   # должно быть зелёным
   ```

6. **Зафиксируй миграцию отдельным MR** в feature-ветке `chore/methodology-bump-to-team-ai-0.2`:
   - Включи в один MR все 4 шага выше.
   - В описании MR — ссылка на этот migration guide.
   - Approve от `@<architects>` (по CODEOWNERS на `.claude/agents/` и `playbooks/`).

## Что НЕ нужно делать

- **Не редактировать `.claude/settings.json` вручную.** Subagent `merge-coordinator` не добавляет новых deny/ask паттернов — `gen-claude-deny.sh` ничего не меняет.
- **Не запускать `merge-coordinator` отдельной командой.** У него нет slash-команды — он вызывается из `/open-mr` и `/full-ahead`.
- **Не пытаться написать в `merge-coordinator`.** Subagent read-only по контракту (как и остальные три); правки делает основная сессия.

## Что меняется в повседневной работе

- При запуске `/open-mr` AI сначала прогонит `merge-coordinator`. Если ветка устарела относительно `origin/main` — будет блокер с предложением `git fetch origin && git rebase origin/main`. Дальше — обычное открытие MR.
- В autonomous mode `/full-ahead` добавился ещё один gate между `security-auditor` и проверкой критерия приёмки. На типовой фиче это +5-15 секунд токенов; при наличии проблем (rebase, отсутствие changelog-фрагмента) — экономит ревьюеру минуты в MR.
