# Миграция team+ai v0.3 → v0.4

Что подтянуть в **уже скопированном проекте** методологии `team+ai`, чтобы получить функциональность v0.4.0 — security hardening (OWASP LLM Top 10 + semgrep + `inbox/` untrusted) и защиту от AI-induced code duplication (jscpd).

**Внимание:** это **отдельная** миграция, не путать с предыдущим [`v0.2-to-v0.3.md`](v0.2-to-v0.3.md) (ci-tiers). Та миграция вводила Fast/Push/Deep уровни и nightly job в `.gitlab-ci.yml`. Эта — расширение содержания тех же скриптов под security и code-clone.

Также включает фикс PATCH-релиза 0.3.0 → 0.3.1 (несостыковка «three-gate review» в AGENTS.md §6), которая для PATCH формально не требует migration guide, но описана здесь для полноты картины.

## Зачем мигрировать

В v0.4.0 закрыты два ADR, выявленные при ревью методологий 2026-05-22 (сравнение с индустриальной SOTA 2024-2026):

1. **[ADR-1500](https://github.com/msamichev/methodology-ai/blob/main/docs/adr/20260522-1500-security-auditor-owasp-llm-top10-and-semgrep.md)** — security-auditor + OWASP LLM Top 10 + semgrep + `inbox/` untrusted.
2. **[ADR-1530](https://github.com/msamichev/methodology-ai/blob/main/docs/adr/20260522-1530-ci-push-code-clone-defense-against-ai-duplication.md)** — code-clone defense (jscpd).

Обоснование — там же. Кратко:
- AI-сгенерированный код массово воспроизводит deserialization-CVE 2025-2026 (pickle/yaml/eval).
- LLM-application'ы (агенты, RAG) нуждаются в отдельной OWASP-карте.
- GitClear 2024: 8x рост дубликатов от AI, -44% YoY доли рефакторинга. В команде дубликаты особенно дороги (рефакторить чужой код после squash merge).
- `inbox/` как папка пользовательских материалов — точка входа prompt injection.

**Без миграции 0.3.x продолжает работать** — оба новых шага опциональны (через `command -v`). Миграция нужна для активации новых проверок.

## Шаги

1. **Установи опциональные инструменты** (рекомендуется на каждой машине команды):

   ```bash
   pip install semgrep   # либо brew install semgrep
   npm install -g jscpd  # либо pip install jscpd-py
   ```

   В команде также имеет смысл добавить semgrep как stage в `.gitlab-ci.yml` после `/adopt-stack` — это отдельная задача release-инженера, не делается через MR в рамках этой миграции.

2. **Перепиши `scripts/ci-push.sh`** копией из свежего seed: добавился semgrep-блок в Шаге 2 и новый Шаг 5 (Code duplication). Если ты после `/adopt-stack` руками подставлял стек-специфичные команды — смерджи аккуратно.

3. **Перепиши `.claude/agents/security-auditor.md`** из свежего seed. Главное:
   - YAML-frontmatter `description` и первое предложение тела ранее ошибочно ссылались на «методологию solo+ai» (наследие копирования). Исправлено на team+ai.
   - Триггерная эвристика «В `/full-ahead`» дополнена LLM-application файловыми паттернами.
   - Добавлена секция «OWASP LLM Top 10 — если проект использует LLM/embeddings в runtime» (10 пунктов LLM01..LLM10).
   - В описании subagent'а уточнено, что находки попадают в MR-комментарии или self-review (team-специфика).

4. **Обнови `AGENTS.md §4.4 «Безопасность»** — добавь пункт про `inbox/` как untrusted input (см. свежий seed, между «Все секреты — через `.env`» и блоком «Производные файлы»).

5. **Обнови `.claude/expected-tools.md`** — в категории `recommended` добавь `semgrep` и `jscpd` с обоснованиями. Точный текст — в свежем seed.

6. **Обнови `playbooks/full-ahead.md`** — в табличке «Три уровня локального CI» строка Push дополнена semgrep и jscpd (см. свежий seed).

7. **PATCH 0.3.0 → 0.3.1 (если ещё не применён):** в `AGENTS.md §6` таблице команд `/full-ahead` и `/autonomous` — заменить «three-gate review» на «four-gate review» (включая `merge-coordinator` subagent, появившийся в v0.2.0). Описание `/full-ahead` синхронизировать с реальной процедурой v0.3.0 (`ci-push.sh → fix loop → commit → push в feature-ветку → подсказка /open-mr`).

8. **Обнови `VERSION`** на `0.4.0` руками.

9. **Прогон `scripts/validate-links.sh`** — должен быть зелёным.

10. **Прогон `scripts/ci-push.sh`** в Phase 0 — должен пройти, semgrep/jscpd пропустятся с подсказками если не установлены.

## Что НЕ нужно делать

- **Не делай semgrep/jscpd обязательными** — сохраняй опциональность для скопированных проектов.
- **Не понижай threshold jscpd** ниже 5% без причины (на legacy-кодах будет много шума → лучше `.jscpd.json`).
- **Не путай v0.3-to-v0.4 (этот документ) с v0.2-to-v0.3.md** — это две разные миграции поверх друг друга.
- **Не делай миграцию, если в команде уже настроен серверный AI-review** (CodeRabbit/BugBot) — security-auditor локально может дублировать с шумом. В этом случае: установи semgrep/jscpd только release-инженеру, не каждому разработчику.
- **Не вычищай OWASP LLM Top 10 секцию из security-auditor**, если проект сейчас не LLM-app — секция начинается с условия «применять только если…», лишних запусков не создаёт.

## Откат

Если semgrep шумит false positives или jscpd блокирует legacy-код:

1. Удалить semgrep-блок и Шаг 5 из `scripts/ci-push.sh` (либо `unset PATH` для них).
2. Полный откат: восстановить `scripts/ci-push.sh`, `.claude/agents/security-auditor.md`, `AGENTS.md §4.4`, `expected-tools.md`, `playbooks/full-ahead.md` из git-истории тега `v0.3.1`. Вернуть `VERSION` на `0.3.1`. Все правки в этом релизе — обратимы.
