# Миграция team+ai v0.2 → v0.3

Что подтянуть в **уже скопированном проекте** методологии `team+ai`, чтобы получить функциональность v0.3.0 — разнесённый локальный CI на три уровня (Fast / Push / Deep) с переносом Deep в GitLab CI nightly и вынесенный self-review playbook к моменту `/open-mr`.

## Зачем мигрировать

В v0.3.0 закрыт [ADR-20260522-1400](https://github.com/msamichev/methodology-ai/blob/main/docs/adr/20260522-1400-team-ai-ci-tiers-with-gitlab-nightly-deep.md) (на базе [ADR-20260518-1432](https://github.com/msamichev/methodology-ai/blob/main/docs/adr/20260518-1432-split-local-ci-into-tiers.md), реализованного в `solo+ai` 0.4.0). Идея:

- Монолитный `scripts/local-ci.sh` из 9 шагов разнесён на три уровня по частоте запуска.
- **Deep-проверки (mutation, SBOM, container scan)** в команде живут как **GitLab CI nightly job** (`security-deep:nightly`), а не на машине каждого разработчика. Локально остаются opt-in для отладки.
- **Self-review checklist** читается **перед `/open-mr`**, а не перед каждым push в feature-ветку. Дополняет `merge-coordinator` subagent (структурные vs смысловые проверки).

Зачем это команде:

- Pre-push цикл укладывается в минуты вместо десятков (mutation/SBOM/container — 10-30 минут на средней базе; умножать на N разработчиков × M итераций = существенная нагрузка).
- Deep-проверки централизованы — единая картина для security/compliance, не зависит от привычек разработчиков.
- Reviewer не ждёт получасовой pipeline на MR (DORA-метрика «время на ревью» страдает, если pipeline тяжёлый).

**Без миграции 0.2.x продолжает работать** — `scripts/local-ci.sh` остался как фасад. Миграция нужна, если хочешь получить новые скрипты, self-review playbook, nightly job и обновлённые playbook'и.

## Предусловия

- Проект на team+ai 0.2.0 (есть `merge-coordinator` subagent — он работает в паре с self-review в новом `/open-mr`).
- Есть `.gitlab-ci.yml` (создаётся в `/init-project` для team+ai с 0.1.0).

## Шаги

1. **Скопируй новые файлы** из свежего `team+ai/`:

   ```bash
   cp /path/to/team+ai/scripts/ci-fast.sh scripts/ci-fast.sh && chmod +x scripts/ci-fast.sh
   cp /path/to/team+ai/scripts/ci-push.sh scripts/ci-push.sh && chmod +x scripts/ci-push.sh
   cp /path/to/team+ai/scripts/ci-deep.sh scripts/ci-deep.sh && chmod +x scripts/ci-deep.sh
   cp /path/to/team+ai/playbooks/self-review.md playbooks/self-review.md
   cp /path/to/team+ai/migrations/v0.2-to-v0.3.md migrations/v0.2-to-v0.3.md
   ```

2. **Перепиши `scripts/local-ci.sh`** из свежего seed. Это тонкий orchestrator (~55 строк) вместо монолита. Внимание: если ты после `/adopt-stack` руками подставлял команды стека в `local-ci.sh` — перенеси их в соответствующие новые скрипты (build/types/SAST/SCA/tests → `ci-push.sh`; mutation/SBOM/container → `ci-deep.sh`).

3. **Обнови `.gitlab-ci.yml`** — добавь блок-комментарий и job `security-deep:nightly`:

   ```yaml
   security-deep:nightly:
     stage: security
     image: alpine:3.20   # /adopt-stack заменит на стек-релевантный образ
     before_script:
       - apk add --no-cache bash
     script:
       - bash scripts/ci-deep.sh
     rules:
       - if: '$CI_PIPELINE_SOURCE == "schedule"'
     allow_failure: false
   ```

   Точное место — после `security:placeholder`, до закомментированных deploy-jobs. Полный текст с комментарием-инструкцией про настройку schedule — в свежем `team+ai/.gitlab-ci.yml`.

4. **Настрой GitLab Pipeline schedule** (операционная задача release-инженера или maintainer'а, **не** делается через MR):

   - GitLab → ваш проект → Settings → CI/CD → Pipeline schedules → New schedule.
   - Description: `Nightly deep security checks`.
   - Interval: `0 3 * * *` (удобное окно низкой нагрузки, UTC).
   - Cron timezone: `UTC`.
   - Target branch: `main`.
   - Activated: ✓.

   Без этой настройки job `security-deep:nightly` не будет запускаться. До `/adopt-stack` он бежит шаблонный `ci-deep.sh` и печатает «нечего проверять» — это норма.

5. **Перепиши `playbooks/full-ahead.md`** — Шаг 2 переключён на `scripts/ci-push.sh`, добавлен self-review в Шаге 6.2 (с уточнением «только если push идёт под `/open-mr`»), переписан раздел «Три уровня локального CI».

   Если ты сам правил этот playbook — не перезаписывай слепо, а смерджи. Логика autonomous mode (Шаги 4.1–4.5 с four-gate review) сохранена.

6. **Перепиши `playbooks/save-all.md`** — Шаг 3 ссылается на `scripts/ci-push.sh`, явно сказано, что self-review здесь **не нужен** (он — pre-MR гейт).

7. **Перепиши `playbooks/open-mr.md`** — добавлен новый **Шаг 3.5 «Self-review (обязательный pre-MR гейт)»**. Прежний Шаг 4 (merge-coordinator) сохраняется, Шаг 2 (pre-flight) обновлён под `ci-push.sh`.

8. **Обнови `AGENTS.md §5 п.10`:**

   Было:
   ```markdown
   10. **Перед push** — `scripts/validate-links.sh` зелёный. `local-ci.sh` зелёный.
   ```

   Стало:
   ```markdown
   10. **Перед push** в feature-ветку — `scripts/validate-links.sh` зелёный, `scripts/ci-push.sh` зелёный. **Перед `/open-mr`** — прочитан `playbooks/self-review.md`. Глубокие проверки (`scripts/ci-deep.sh`: mutation, SBOM, container scan) — opt-in локально и nightly job `security-deep:nightly` в GitLab CI (см. `.gitlab-ci.yml`).
   ```

9. **Обнови `METHODOLOGY.md §5 «Качество — автоматикой»` и `§9 «Качество кода»`** — оба раздела переписаны под три уровня с указанием «где живёт» (локально vs GitLab CI nightly). Точный текст — в свежем seed.

10. **Обнови три `SKILL.md` description**:
    - `.claude/skills/full-ahead/SKILL.md` — упоминает `ci-push.sh`, self-review только перед `/open-mr`, four-gate review, Deep в CI nightly.
    - `.claude/skills/save-all/SKILL.md` — упоминает `ci-push.sh` как pre-push требование; self-review явно вынесен в `/open-mr`.
    - `.claude/skills/open-mr/SKILL.md` — упоминает обе pre-MR gate: self-review (semantic) + merge-coordinator (structural).

    Это критично для авто-триггера в Claude Code — без обновления Claude может запустить старый flow.

11. **Обнови `playbooks/README.md`** — добавь строку `self-review.md` в таблицу playbook'ов; уточни описания `full-ahead.md`, `save-all.md`, `autonomous-mode.md` (four-gate review вместо three-gate). Точный текст — в свежем seed.

12. **Убедись, что pre-commit hook установлен** (одноразово на каждой машине):

    ```bash
    pre-commit install --hook-type pre-commit --hook-type commit-msg
    ```

    Без этого Fast-уровень (`ci-fast.sh`) не отработает на самом commit, и придётся прогонять `scripts/local-ci.sh --fast` руками перед push.

13. **Обнови `VERSION`** на `0.3.0` руками.

14. **Прогон `scripts/validate-links.sh`** — должен быть зелёным.

15. **Прогон `scripts/local-ci.sh`** (без флагов) — эквивалент `ci-push.sh`, должен пройти в Phase 0 с понятными сообщениями.

## Что НЕ нужно делать

- **Не удаляй `scripts/local-ci.sh`** — он остался как фасад. CI-пайплайны и документация ссылаются на этот путь.
- **Не клади mutation/SBOM/container scan обратно в `ci-push.sh`** — это нивелирует смысл миграции. Хочешь прогнать всё разом локально перед редким MR — есть `scripts/local-ci.sh --deep`.
- **Не передавай `--skip-mutation` в новый `local-ci.sh`** — флаг устарел (mutation теперь только в Deep, и не запускается по умолчанию). Скрипт напечатает warning и продолжит.
- **Не настраивай Pipeline schedule через MR** — это GitLab UI-операция в Settings → CI/CD → Pipeline schedules. Запиши в `retrospectives/YYYY-MM-DD-release-vX.Y.Z.md`, что настройка сделана и кем.
- **Не делай self-review обязательным перед каждым push в feature-ветку** — это противоречит ADR-20260522-1400 (feature-checkpoint push — это checkpoint, а не финал; self-review нужен на этапе `/open-mr`, когда автор смотрит на свой MR глазами ревьюера).
- **Не удаляй секции `[0.1.x]` и `[0.2.x]` в CHANGELOG.md** — история должна оставаться целой.

## Откат

Маловероятно, но если разнесение по уровням окажется неудобным:

1. Восстанови старый `scripts/local-ci.sh` из git-истории (соответствующий тег `v0.2.0` в репозитории методологии или из своего git).
2. Удали `scripts/ci-fast.sh`, `ci-push.sh`, `ci-deep.sh`.
3. Удали job `security-deep:nightly` из `.gitlab-ci.yml` (или отключи GitLab schedule).
4. Восстанови старые версии `playbooks/full-ahead.md`, `save-all.md`, `open-mr.md`, `AGENTS.md §5 п.10`, `METHODOLOGY.md §5/§9`.
5. `playbooks/self-review.md` можно оставить — он не активен без явных ссылок из других playbook'ов.
6. Верни `VERSION` на `0.2.0`.

Откат **обратимый** — содержательно ничего не теряется, кроме фокусированных гейтов.
