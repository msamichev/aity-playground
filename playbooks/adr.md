# Adr Playbook

> **Универсальная процедура.** Читается любым AI: Claude, Codex, DeepSeek, Minimax, Cursor, Gemini, Aider и т.п.
> **Триггер:** `/adr` или соответствующая фраза на естественном языке (см. таблицу триггеров в [AGENTS.md](../AGENTS.md)).
> **Тонкая обёртка для Claude Code:** [.claude/skills/adr/SKILL.md](../.claude/skills/adr/SKILL.md).

---

Создаёт новый ADR.

## Когда использовать

ADR пишется для любого решения, на которое через 6 месяцев кто-то спросит «почему так?». Примеры:
- Выбор БД, фреймворка, библиотеки
- Подход к авторизации, кэшированию, миграциям
- Архитектурный паттерн (CQRS, event sourcing, ...)
- Способ структурирования модулей
- Контракт API

**НЕ для** ADR:
- Решения уровня «как назвать переменную»
- Идейные продуктовые решения, не имеющие технических последствий (это `docs/idea/01-idea.md` или `03-principles.md`)
- Временные эксперименты (`experiments/`)

## Алгоритм

1. Сгенерировать timestamp: `$(date +%Y%m%d-%H%M)`.
2. **Спроси про привязку к задаче** во внешнем трекере (опционально для ADR, в отличие от плана):
   - Если ADR связан с задачей — пользователь даёт ID (например, `PROJ-142`). Префикс читай из `AGENTS.md §0` поле `id_prefix`.
   - Если ADR обзорный / архитектурный без привязки (выбор БД, общий паттерн) — без ID.
3. Принять slug от пользователя (kebab-case): «choose-postgres», «use-langgraph», и т.д.
4. Имя файла:
   - С ID: `docs/adr/YYYYMMDD-HHmm-<ID-NNN>-${slug}.md` (например, `20260522-1430-PROJ-142-google-oauth.md`).
   - Без ID: `docs/adr/YYYYMMDD-HHmm-${slug}.md`.
5. Скопировать `docs/adr/template.md`.
6. Заполнить заголовок и дату. Если есть ID — добавь в шапке ADR ссылку на задачу в трекере (`<tracker_url>/<ID>`).
7. Спросить пользователя по очереди:
   1. **Context** — какую проблему решаем, какие ограничения?
   2. **Decision** — что именно решили?
   3. **Alternatives considered** — какие варианты были, почему отказались?
   4. **Consequences** — что хорошо/плохо/что тянет за собой?
   5. **Связки** — каких документов касается?
8. **Индекс ADR не редактируй вручную.** Он генерируется скриптом [`scripts/build-adr-index.py`](../scripts/build-adr-index.py) на pre-commit из директории `docs/adr/` (см. [AGENTS.md §3](../AGENTS.md#3-индекс-adr)). В team+ai ручная таблица в AGENTS.md упразднена — заменена на `docs/adr/INDEX.md`.
9. **Не коммитить автоматически.** Пользователь решает когда коммитить (через MR в feature-ветке).

## Правила

- `Status` нового ADR — обычно `Accepted` (если решение принято) или `Proposed` (если идёт обсуждение).
- При **отмене старого решения** — НЕ редактируй старый ADR. Создай новый, в старом меняй только `Status: Superseded by <new-adr>`.
- `Alternatives considered` — обязательно. «Ничего не рассматривали» — признак слабого решения.
- ADR должен быть **читаем через 6 месяцев** — расписывай контекст подробно.
