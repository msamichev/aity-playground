# Skills Audit Playbook

> **Универсальная процедура.** Читается любым AI: Claude, Codex, DeepSeek, Minimax, Cursor, Gemini, Aider и т.п.
> **Триггер:** `/skills-audit` или фразы «аудит skills», «какие skills я не использую», «почисти skills».
> **Тонкая обёртка для Claude Code:** [.claude/skills/skills-audit/SKILL.md](../.claude/skills/skills-audit/SKILL.md).

---

## Зачем

Каждый установленный skill **тратит токены** (его описание загружается в контекст). 8-12 skills — оптимум. Больше — налог на контекст.

Skills, которые установлены, но не использовались за последние 30+ дней — кандидаты на удаление.

## Когда запускать

- **Раз в месяц** — рекомендуется. Можно встроить в `/week-plan` итоги первой недели месяца.
- **Когда заметил, что Claude путается** между skill'ами или загрузки кажется медленной.
- **Перед добавлением новых skills** — освободить место.

## Алгоритм

### Шаг 1. Список установленных skills

```bash
claude plugin list 2>/dev/null
ls ~/.claude/skills/ 2>/dev/null         # personal
ls .claude/skills/ 2>/dev/null           # project
```

Собрать в общий список.

### Шаг 2. Проверить активность

Это самая сложная часть — у Claude Code **нет встроенной телеметрии** «когда последний раз триггерился skill».

**Возможные источники сигнала:**

1. **Git-история проекта** — если skill создавал файлы или менял документы, их следы в коммитах.
2. **Файлы лога Claude Code** (`~/.claude/logs/` или похожее, зависит от установки).
3. **Спросить пользователя напрямую** — «вспомни, использовал ли ты `frontend-design` за последний месяц?»

**Простая эвристика (без точных данных):**

| Скилл | Признак активного использования |
|---|---|
| `frontend-design` | Недавние правки в `src/components/`, новые .tsx/.vue/.svelte файлы |
| `react-best-practices` | Аналогично |
| `web-design-guidelines` | Сообщения в коммитах про accessibility, performance UI |
| `webapp-testing` | Файлы в `e2e/`, `tests/e2e/`, или папке с Playwright-тестами |
| `docx`/`xlsx`/`pdf`/`pptx` | Скрипты или функции, генерирующие соответствующие документы |
| `mcp-builder` | Папка с MCP-серверами или конфиги для них |
| `Superpowers` | Файлы из его flow: `brainstorm-notes.md`, `plan.md` в специфичной структуре |

### Шаг 3. Сформировать отчёт

Структура:

```
📊 Skills Audit за последний месяц

Установлено: 8 skills

🟢 Активно используются:
  - frontend-design (правки UI, 12 коммитов)
  - react-best-practices (4 коммита с performance-улучшениями)
  - webapp-testing (7 правок в e2e/)

🟡 Использовались редко:
  - web-design-guidelines (1 коммит про accessibility)
    → Оставить, но проверить: возможно, не активировался когда нужно?

🔴 Не использовались (кандидаты на удаление):
  - docx (нет коммитов в коде, работающем с .docx)
  - pdf (то же)
    → Если функционал генерации документов не нужен — удалить:
      /plugin uninstall docx@anthropic-agent-skills
      /plugin uninstall pdf@anthropic-agent-skills

🤔 Неоднозначно:
  - Superpowers
    → Сложно определить: используется ли brainstorm+write-plan+execute-plan?
    → Спрашиваю напрямую: ты в этом месяце пользовался /superpowers:brainstorm?

📈 Статистика:
  - Активных skills: 3 / 8 (37%)
  - Если контекст-бюджет узкий, рассмотри удаление красных
```

### Шаг 4. Не удалять автоматически

**Только рекомендации.** Удаление инструмента — это решение пользователя. Покажи команды, он копирует и запускает (или нет).

### Шаг 5. Записать в ретро или в журнал

Если решение про удаление принято — это тоже момент для записи. Можно через `/retro` или короткую запись в специальный файл `.claude/skills-changelog.md`:

```markdown
# Skills changelog

## 2026-05-17
- Удалены: docx, pdf (не использовались 30+ дней)
- Установлены: composition-patterns (для нового рефакторинга UI)
```

Это **необязательно**, но полезно для будущего `/skills-audit` — увидеть «когда что было».

## Что НЕ делает

- **Не удаляет skills автоматически.** Только рекомендации с командами.
- **Не критикует выбор пользователя.** Если он хочет оставить «на всякий случай» — это его право.
- **Не предполагает 100% точность.** Без встроенной телеметрии аудит эвристический.
