# Delegate Playbook

> **Универсальная процедура.** Читается любым AI: Claude, Codex, DeepSeek, Minimax, Cursor, Gemini, Aider и т.п.
> **Триггер:** `/delegate` или фразы «раздай ботам», «делегируй команде», «запусти ботов на …» (см. таблицу триггеров в [AGENTS.md](../AGENTS.md)).
> **Тонкая обёртка для Claude Code:** [.claude/skills/delegate/SKILL.md](../.claude/skills/delegate/SKILL.md).

---

`/delegate` — **maintainer-сторона профиля исполнителя «автономный бот»**. Умная (дорогая) главная сессия мейнтейнера берёт большую задачу, **правильно декомпозирует** её на дизъюнктные подзадачи и **раздаёт** их флоту дешёвых headless-ботов (DeepSeek/Qwen) в изолированные песочницы. Боты автономно ведут свои feature-ветки → проходят гейты → открывают PR. **Ревью и мерж этих PR остаются за мейнтейнером** — это финальный гейт team+ai, и `/delegate` его не трогает.

В терминах ролей team+ai: **боты = команда, ты = мейнтейнер** (см. [мета-ADR 20260613-0008](https://github.com/msamichev/methodology-ai/blob/main/docs/adr/20260613-0008-team-ai-autonomous-cheap-bot-executors.md)). `/delegate` — операционализация этого профиля со стороны того, кто раздаёт задачи. Обоснование самой команды — [мета-ADR 20260616-0144](https://github.com/msamichev/methodology-ai/blob/main/docs/adr/20260616-0144-team-ai-delegate-fan-out-to-bot-fleet.md).

Допустимость дешёвого автономного исполнителя держится на **контракте безопасности**, который нельзя ослаблять: **(а)** защищённый `main` — бот не мержит; **(б)** CI/гейты как **ground-truth** — доверяем факту (зелёный CI), а не прозе бота «готово»; **(в)** человек-мейнтейнер на финальном гейте. `/delegate` доводит до PR и собирает статус — мерж всегда за человеком.

## Граница ответственности (не путать команды)

| Команда | Профиль | Роль |
|---|---|---|
| `/autopilot` | **одно-агентный**, в сессии | Автономный обход **своего** многофазного плана **в своей feature-ветке**, коммит на фазу, без push. |
| `/full-ahead` | один цикл | `ci-push.sh` + автофикс + four-gate + **push в свою feature-ветку**. |
| **`/delegate`** | **multi-агентный**, внешний флот | Декомпозиция большой задачи → раздача N **внешним** ботам в песочницы → N PR → dashboard. Ревью/мерж — мейнтейнеру. |

`/delegate` **переиспускает** существующие кирпичи: [`/plan`](plan.md) (план/контракты), [`/adr`](adr.md) (фиксация контрактов значимого решения), [`/open-mr`](open-mr.md) + [`/self-review`](self-review.md) (ревью PR ботов мейнтейнером). Новой логики мержа не вводит.

## Когда использовать

- Есть **большая** задача, которую можно честно разрезать на **узкие, независимые** куски (каждый кусок — своя пара/группа новых файлов, общие точки не трогаются).
- Под рукой развёрнут `agent-aity` (или совместимый headless-инструмент) — путь к нему в env `AITY_HOME`.
- Ты — мейнтейнер и готов ревьюить N PR после прогона.

## Когда НЕ использовать

- **Тривиальная задача** (правка в одном-двух файлах) — дешевле сделать самому + `/save`. Делегирование флоту имеет overhead (декомпозиция + N контейнеров + N ревью).
- **Переплетённая задача**, где подзадачи делят файлы и не разрезаются на дизъюнктные — **секвенс** (по одной) или **вообще не дроби**; параллелить нельзя.
- **Сильная неопределённость требований**, которую нельзя снять одним блоком вопросов на старте, — сначала `/plan`, потом, может быть, `/delegate`.
- `AITY_HOME` не задан / инструмент не развёрнут — останови с инструкцией, не запускай.

## Рубрика декомпозиции (суть команды)

Опирается на проверенные находки (Anthropic multi-agent; исследования параллельных кодинг-агентов). Это **не «суждение модели» в каждом прогоне**, а фиксированная процедура — пройди её по пунктам.

### 1. Контракт-first

**Сначала зафиксируй интерфейсы между подзадачами** — сигнатуры функций/эндпоинтов, форматы данных, имена общих типов/модулей, контракты API. Запиши их в план ([`/plan`](plan.md)) или, если развилка значима, в ADR ([`/adr`](adr.md)). Боты должны реализовывать **против общего зафиксированного контракта**, а не угадывать интерфейс соседа — каждый бот изолирован и не видит код/контекст другого.

> Без контракта-first два бота независимо «придумают» несовместимые сигнатуры на стыке — и стык не соберётся при мерже, даже если файлы дизъюнктны.

### 2. Карта владения файлами

Для **каждой** подзадачи перечисли явно: **какие файлы она трогает** (создаёт/меняет). Затем построй карту:

- Подзадачи с **дизъюнктными** наборами файлов → можно **параллелить**.
- Подзадачи, **делящие хотя бы один файл** → **НЕ параллелить**, секвенировать (одна за другой) либо переразрезать задачу.

**Дизъюнктность файлов — единственная защита от merge-конфликтов** (правило team+ai [«один файл = одна запись»](../AGENTS.md#43-git), распространённое на ботов в [20260613-0008](https://github.com/msamichev/methodology-ai/blob/main/docs/adr/20260613-0008-team-ai-autonomous-cheap-bot-executors.md)). Общие точки сборки (`app.ts`, `routes`, `store`, DI-контейнер, миграции) — частый скрытый shared-файл: если две подзадачи обе регистрируются там, они **не дизъюнктны**. Вынеси регистрацию в отдельную секвенс-подзадачу или сделай сам после мержа.

### 3. Потолок параллелизма 3–5 воркеров на репо

Не запускай больше **3–5** ботов одновременно на одном репозитории, даже при изоляции песочниц. Дальше растёт вероятность конфликтов на общих точках и ревьюерская нагрузка превышает ёмкость мейнтейнера. Больше кусков — гони **волнами** (3–5, смержил, следующая волна).

### 4. Правило масштаба (явное, не на «суждение»)

| Размер задачи | Что делать |
|---|---|
| **Тривиально** (1–2 файла, понятно как) | **Не делегируй** — сделай сам + `/save`. |
| **Узкие независимые куски** (каждый = своя группа файлов) | Делегируй **2–5 ботам** параллельно. |
| **Всё переплетено** (общие файлы, сильная связность) | **Секвенс** (по одной) или **не дроби** вовсе — делай сам/`/autopilot`. |

### 5. Самодостаточный промпт на подзадачу

Бот **не наследует контекст** главной сессии — стартует с нуля в чистом клоне. Поэтому промпт каждой подзадачи самодостаточен и содержит:

- **Цель** — что должно появиться (одно-два предложения, наблюдаемо).
- **Затрагиваемые файлы** — какие создать/изменить (из карты владения, п.2). Явно: «общие `<...>` не трогать».
- **Контракт** — сигнатуры/интерфейсы со стыка (из п.1), которым обязан соответствовать результат.
- **Критерий готовности** — машинно-проверяемо: какие тесты зелёные, какой эндпоинт что отвечает, что CI зелёный (`ground-truth`, не «я сделал»).

Промпт пишется так, будто читателю **не доступно** ничего, кроме репозитория и этого текста.

## Контракт запуска (через `AITY_HOME`)

Skill **не хардкодит** путь к инструменту — локализует его через env `AITY_HOME` (путь к чекауту `agent-aity` или совместимого headless-инструмента).

```bash
AITY_HOME="${AITY_HOME:?Задай AITY_HOME — путь к чекауту agent-aity}"
```

Если `AITY_HOME` не задан или скриптов нет — **остановись** и попроси пользователя задать его / развернуть инструмент. Не пытайся угадывать путь.

**Подготовка.** Запиши каждую подзадачу в **отдельный** `.md`-файл (самодостаточный промпт из п.5) — в `tasks/queue/` целевого репо или во временный каталог.

**Параллельный dispatch (основной путь):**

```bash
QUEUE_CONCURRENCY=<n> "$AITY_HOME/scripts/queue.sh" <repo> <backend> <каталог-или-task-файлы...>
```

`queue.sh` сам делает стаггер стартов (анти-коллизия RUNID), гоняет `run.sh` на каждую подзадачу и в конце зовёт `dashboard.sh`. `<n>` — потолок параллелизма (см. рубрику п.3: 3–5).

**Один прогон** (если подзадача всего одна или гонишь секвенс):

```bash
"$AITY_HOME/scripts/run.sh" <repo> "<задача>" <backend>
```

**Сбор статуса** (вердикты, cost, ссылки на PR):

```bash
"$AITY_HOME/scripts/dashboard.sh"
```

`<backend>` = `deepseek` (дефолт) | `qwen` | … — по тому, что развёрнуто в инструменте.

> **Провижн клона.** Сам инструмент (`run.sh`/`queue.sh`) клонирует репо в изолированную песочницу и материализует gitignored-конфиг (хуки + allowlist разрешений), который в свежий клон не доезжает (follow-up [20260613-0008](https://github.com/msamichev/methodology-ai/blob/main/docs/adr/20260613-0008-team-ai-autonomous-cheap-bot-executors.md)). `/delegate` на это полагается — отдельно провижн не зовёт.

## Поток (по шагам)

### Шаг 1. Уточни и подтверди задачу

Прочитай задачу. Если требований не хватает — задай **все** уточняющие вопросы **одним блоком** (механика front-loaded clarification — [plan playbook](plan.md) Шаг 2) **до** запуска. Дорогой fan-out не начинаем с неполным пониманием.

### Шаг 2. Построй план/ADR декомпозиции

Пройди **рубрику декомпозиции** целиком и зафиксируй результат:

1. **Контракты** между подзадачами (п.1) — в план или ADR.
2. **Карта владения файлами** (п.2) — таблица «подзадача → файлы», с пометкой **параллельно / последовательно** для каждой.
3. **Список подзадач** с самодостаточными промптами (п.5).
4. Проверь **потолок** (п.3, ≤3–5) и **правило масштаба** (п.4) — может, часть кусков делегировать не стоит.

Оформление: используй [`/plan`](plan.md) для плана задачи; для значимой архитектурной развилки на стыке контрактов — [`/adr`](adr.md).

### Шаг 3. Покажи план и дождись подтверждения

Покажи пользователю: контракты + карту файлов (что параллельно, что секвенс) + список подзадач + выбранный `<backend>` и `<n>`. **Дождись явного «ок».** Молча дорогой fan-out не запускаем (N контейнеров = N трат).

Формат блока на подтверждение:

```
🤖 Готов раздать ботам задачу «<название>».

Контракты (стыки): <кратко>.
Параллельно (дизъюнктные файлы):
  • <подзадача A> → файлы: <...>
  • <подзадача B> → файлы: <...>
Последовательно (делят файлы):
  • <подзадача C> после <A> → файлы: <...>

Backend: <deepseek|qwen>. Параллелизм: <n> (≤5).
Запускаю? (да / правки / отмена)
```

### Шаг 4. Запиши task-файлы и запусти dispatch

После «ок»: запиши каждую подзадачу в свой `.md` (Шаг подготовки), затем вызови `queue.sh` через `AITY_HOME` (параллельный путь) либо `run.sh` для секвенса/одиночки. Дизъюнктные — в одну волну ≤`<n>`; делящие файлы — следующей волной после мержа предыдущей.

### Шаг 5. Покажи dashboard и перечисли PR

По завершении прогона вызови `dashboard.sh`. Покажи пользователю сводку: по каждому боту — вердикт, cost, ссылка на PR. Перечисли открытые PR явно.

```
✅ Флот отработал задачу «<название>».
  • <runid A> — ✅ зелёный CI — PR <url>
  • <runid B> — ✅ зелёный CI — PR <url>
  • <runid C> — ⚠ требует внимания — <что не так>

Дальше (твой шаг как мейнтейнера): ревью каждого PR → /self-review → мерж (gh pr merge / glab mr merge).
/delegate НЕ мержит — это финальный гейт team+ai за тобой.
```

### Шаг 6. Стоп. Ревью и мерж — отдельный шаг мейнтейнера

`/delegate` **останавливается на PR**. Ревью и мерж — существующие флоу [`/open-mr`](open-mr.md) / [`/self-review`](self-review.md) + `gh pr merge` / `glab mr merge`, которые делает **человек**. Команда их не вызывает и не мержит сама.

## Гайдрейлы (обязательны)

- **Мерж в защищённый `main` — только человек.** Бот мержить не может (protected `main`); full issue→PR→auto-merge индустрией не считается безопасным в 2026. `/delegate` доводит до PR, **не дальше**.
- **CI/гейты — ground-truth, не проза бота.** Готовность подзадачи — по **факту** (зелёный CI, прошедший тест из критерия готовности), а не по самооценке бота «сделано». Слабая модель систематически переоценивает свою работу.
- **Дизъюнктность файлов — единственная защита от конфликтов.** Нарушил (подзадачи делят файл) → **секвенс**, не параллель. Скрытый shared-файл (общая точка регистрации) ломает дизъюнктность так же, как явный.
- **Потолок 3–5 воркеров** на репо. Больше — волнами, а не одним залпом.
- **Правило масштаба явное.** Тривиально → сам; узкие независимые → 2–5 ботов; переплетённое → секвенс/не дроби. Не отдавай флоту **широкую** задачу — валидированы только узкие.
- **Дешёвые модели слабее в длинных циклах.** Поэтому: узкие задачи + самодостаточный промпт + финальный гейт мейнтейнера — критичны. Длинный автоном слабой моделью не валидирован.
- **`AITY_HOME`, не хардкод.** Нет env / инструмента → стоп с инструкцией.
- **Дорогой fan-out — только после подтверждения** (Шаг 3). N контейнеров запускаются осознанно.

## Совместимость с другими AI

Процедура (рубрика + поток + контракт запуска) — текст, читается любым AI. Сам fan-out требует развёрнутого headless-инструмента (`agent-aity` или совместимого) и `AITY_HOME`. В Claude Code `/delegate` — skill с авто-триггером; другие AI читают этот playbook по таблице в `AGENTS.md`.
