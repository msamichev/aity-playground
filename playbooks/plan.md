# Plan Playbook

> **Универсальная процедура.** Читается любым AI: Claude, Codex, DeepSeek, Minimax, Cursor, Gemini, Aider и т.п.
> **Триггер:** `/plan` или соответствующая фраза на естественном языке (см. таблицу триггеров в [AGENTS.md](../AGENTS.md)).
> **Тонкая обёртка для Claude Code:** [.claude/skills/plan/SKILL.md](../.claude/skills/plan/SKILL.md).

---

Создаёт новый план фичи. **Это точка входа в фазу `phase-2-feature`** — перед созданием плана прогоняем pre-feature чек окружения.

## Алгоритм

### Шаг 0. Pre-feature чек окружения

Прежде чем создавать план — прогони `/doctor` (см. [doctor playbook](doctor.md)). Это пред-фазная проверка перед началом новой фичи в Steady state.

**Когда тихо пропустить:** если в `.claude/settings.local.json` есть `doctorAcknowledged: true` И `doctorLastPhase == "phase-2-feature"` И `doctorLastRun` свежий (в пределах текущей сессии) — Шаг 0 пропускается без вывода. Иначе — выполняется.

**Что делает `/doctor` здесь:** определит фазу `phase-2-feature` (если эта команда `/plan` создаст активный план — это и есть фаза), проверит required + recommended + contextual + phase-2-feature инструменты и skills, найдёт конфликты.

**Реакции на вывод:**

- **Всё зелёное / только optional пропущены** → переходи к Шагу 1 без диалога.
- **Отсутствуют `required` инструменты** → спроси пользователя одним сообщением:
  > Не хватает: `<список required>`. Без этого `/full-ahead` упадёт. Продолжаем создание плана или сначала установишь? (продолжать / установить / отмена)
  Не блокируй автоматически — это его выбор.
- **Найдены конфликты** → выведи их перед созданием плана с пометкой «не блокирующее, но стоит обратить внимание». Переходи к Шагу 1.

После Шага 0 — обнови `doctorLastPhase` и `doctorLastRun` в `.claude/settings.local.json`.

### Шаг 1. Создание плана

1. **Спроси ID задачи из внешнего трекера** (например, `PROJ-142`). Префикс читай из `AGENTS.md §0` поле `id_prefix`. Если задачи нет — спроси, не нужно ли её сначала завести в трекере; план без ID допустим только для технических задач (`chore`) или мета-улучшений методологии.
2. Принять `<slug>` от пользователя (если не дан — спросить, kebab-case).
3. Сгенерировать имя:
   - С ID: `plans/$(date +%Y-%m-%d)-<ID-NNN>-${slug}.md` (например, `plans/2026-05-22-PROJ-142-google-oauth.md`).
   - Без ID (редко, для chore): `plans/$(date +%Y-%m-%d)-${slug}.md`.
4. Скопировать `plans/TEMPLATE.md` в новый файл.
5. Заменить заголовок `<название-фичи>` на читаемое название. В шапке плана — обязательно ссылка на задачу в трекере: `<tracker_url из AGENTS.md §0>/<ID>`.
6. **Не заполнять секции** — это работа пользователя в диалоге.
7. Переходи к Шагу 2 (front-loaded clarification) — **не** задавай вопросы по одному.

### Шаг 2. Front-loaded clarification

**Цель:** собрать все уточняющие вопросы **одним блоком до начала работы**, а не возвращаться к пользователю в середине реализации. Это снижает количество интеракций — главная цель autonomous mode (см. [autonomous-mode playbook](autonomous-mode.md)).

**В autonomous mode (`autonomous_mode: enabled` в AGENTS.md §0) — обязательно.**
**Вне autonomous mode — рекомендуется**, но допустима частичная декомпозиция (диалог по фазам).

Алгоритм:

1. Прочитай задачу пользователя.
2. Прочитай существующие `docs/idea/`, `docs/architecture/`, `docs/adr/` — что уже решено.
3. **Сгенерируй полный список открытых вопросов**, разбитый по категориям:
   - **Цель и критерий приёмки.** Что должно быть к концу? По каким наблюдаемым критериям поймём, что готово?
   - **Скоуп.** Что внутри, что вне. Какие модули затронуты.
   - **Архитектура.** Развилки, на которые нужно ответить (если есть). Конфликты с существующими ADR.
   - **Внешние интеграции.** Какие API, библиотеки, сервисы. Версии. Лимиты.
   - **Безопасность.** Чувствительные данные, auth flows, secret handling.
   - **Тесты.** Какие классы тестов нужны (unit / integration / e2e). Какие edge-cases обязательны.
   - **Не-функциональное.** Производительность, доступность, локализация — если применимо.
   - **Предположения.** Что я предполагаю и хочу подтвердить.
4. Задай **все вопросы одним сообщением**. Не возвращайся за уточнениями по ходу — это нарушает контракт autonomous mode.
5. Жди ответа. После ответа — Шаг 3.

**Исключения**, когда допустим возврат с уточнением:
- Внешний блокер (упал MCP, нет доступа к репозиторию, отсутствует обязательный инструмент).
- Принципиальное противоречие, которое раскрылось только в коде (с конкретным file:line и предложением 2-3 вариантов решения).
- Пользователь сам сказал в плане «остановись на шаге X и спроси».

### Шаг 3. Формализация критерия приёмки

В файле `plans/YYYY-MM-DD-<slug>.md` секция «Критерий приёмки» заполняется в **машиночитаемом виде** — markdown checklist:

```markdown
## Критерий приёмки

- [ ] <конкретно проверяемое утверждение 1, например: «GET /auth/github возвращает 302 на github.com/login/oauth/authorize с state-параметром»>
- [ ] <утверждение 2, например: «токены хранятся в httpOnly cookie с SameSite=Lax»>
- [ ] <утверждение 3, например: «тест tests/test_oauth.py::test_csrf_state_validation зелёный»>
- [ ] <утверждение 4, например: «code-reviewer subagent вернул ОДОБРЕНО»>
- [ ] <утверждение 5, например: «security-auditor subagent вернул ОДОБРЕНО»>
```

Правила:
- Каждый пункт — **наблюдаемое, проверяемое утверждение**, не «реализовать X».
- Минимум 3 пункта. Если задача крупная — 5-8 пунктов.
- В autonomous mode критерий приёмки обязателен — без него `/full-ahead` не сможет понять, когда цикл завершён.
- В обычном режиме критерий приёмки желателен, но допустимо начать без него и дописать по ходу.

После заполнения секций «Цель», «Контекст», «Фазы», «Критерий приёмки» — план готов. Дальше пользователь либо переходит к реализации (вручную или через `/full-ahead`), либо запускает `/critic` для стресс-теста плана, либо отдаёт план на автономное выполнение через [`/autopilot`](autopilot.md) — тот пройдёт по `## Фазы` в твоей feature-ветке, коммитя каждую фазу с `Refs:` (без push), и допишет секцию `## Журнал автономного прогона`.

## Правила

- **Один план = одна функция.** Если задача слишком крупная — разбей на несколько планов или сделай родительский «эпик» с ссылками.
- Если уже есть открытый план той же сессии — спроси «продолжаем этот или начинаем новый».
- После создания плана — **не пиши код**. Сначала проговорите фазы и критерии готовности.
- **Шаг 0 нельзя пропустить вручную без причины.** Если хочется обойти — лучше использовать `/doctor` отдельно и поставить `doctorAcknowledged: true`, чтобы `/plan` пропустил его сам.
- **Шаг 2 обязателен в autonomous mode.** Возврат с уточнениями по ходу реализации — нарушение контракта.
