# Sync Idea Playbook

> **Универсальная процедура.** Читается любым AI: Claude, Codex, DeepSeek, Minimax, Cursor, Gemini, Aider и т.п.
> **Триггер:** `/sync-idea` или соответствующая фраза на естественном языке (см. таблицу триггеров в [AGENTS.md](../AGENTS.md)).
> **Тонкая обёртка для Claude Code:** [.claude/skills/sync-idea/SKILL.md](../.claude/skills/sync-idea/SKILL.md).

---

Подтягивает новые материалы из `inbox/` и недавних обсуждений в `docs/idea/` и (если применимо) `docs/architecture/`.

В отличие от `/init-project` (запускается **один раз**), эта команда — **повторяемая**.

## Алгоритм

### Шаг 1. Что нового

```bash
# Файлы в inbox/, появившиеся после последнего sync
# Эвристика: смотрим mtime, или явный маркер `inbox/.last-sync`
```

Если есть маркер `inbox/.last-sync` — берёшь файлы новее него.
Если нет — спрашиваешь у пользователя «всё с прошлого раза или что-то конкретно?».

Дополнительно: если в текущей сессии было содержательное обсуждение, не отражённое в файлах — учитывай и его.

### Шаг 2. Сопоставление с разделами

Прочитай новые материалы. Для каждого определи, к какому разделу `docs/idea/` это относится:

| Содержимое нового материала | Целевой раздел |
|---|---|
| Что/для кого/зачем, ядро идеи, принятые продуктовые решения | `docs/idea/01-idea.md` |
| Конкуренты, что не сработало у других, рыночный контекст | `docs/idea/02-research.md` |
| Принципы (что важно делать так / не делаем) | `docs/idea/03-principles.md` |
| MVP-объём, capabilities, этапы развития | `docs/idea/04-mvp.md` |
| Риски | `docs/idea/05-risks.md` |
| Долгосрочная картинка (1-3 года) | `docs/product/vision.md` |
| Сегменты, путь пользователя, возражения | `docs/product/audience.md` |
| Грубая высокоуровневая архитектура | `docs/architecture/overview.md` |

### Шаг 3. Slot-filling (НЕ перезапись)

Для каждого targeted раздела:
1. Прочитай существующий файл.
2. Извлеки из нового материала ту информацию, которой нет в файле.
3. **Дополни**, не перезаписывай. Существующие утверждения не трогай.
4. Если новый материал **противоречит** существующему утверждению — **остановись и спроси**:
   > В `docs/idea/03-principles.md` написано «не продаём в B2C», но в новом материале говорится «рассматриваем B2C-пилот». Это пересмотр решения или контекст другой?
5. Если пересмотр — предложи создать ADR (через `/adr`), чтобы зафиксировать изменение мысли. **Не редактируй idea-файл напрямую** — это снимок, должен быть стабилен.

### Шаг 4. Архитектурные намёки → docs/architecture/

Если в новых материалах есть **техническая** конкретика (компоненты, потоки, интеграции, выбор технологий) — это идёт **не** в `docs/idea/`, а в `docs/architecture/`. Применяй ту же логику slot-filling.

### Шаг 5. Маркер последней синхронизации

```bash
touch inbox/.last-sync
```

Это позволяет следующему `/sync-idea` быстро определить «что новое».

### Шаг 6. Валидация ссылок

```bash
scripts/validate-links.sh
```

Все секции `## Связки` обновлённых файлов должны быть валидны.

### Шаг 7. Отчёт

Покажи пользователю **краткое сравнение «было / стало»** по каждому изменённому файлу — 1-2 предложения.

## Правила

- `docs/idea/` — **frozen snapshot**. Изменения внутри файлов — только дополнения, не перезапись.
- Противоречия — повод создать ADR, а не править idea.
- Если в `inbox/` пришёл материал, который не вписывается ни в idea, ни в architecture (например, новые продуктовые данные) — спроси, куда положить: `docs/product/` или новая категория.
