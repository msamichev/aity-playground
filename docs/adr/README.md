# Architecture Decision Records

Каждое значимое архитектурное или техническое решение фиксируется отдельным ADR-файлом.

Layout: [MADR 4.0](https://adr.github.io/madr/) с расширением (наш `Consequences` детализирован Positive / Negative / Follow-ups).

## Имена файлов

`YYYYMMDD-HHmm-kebab-case-slug.md` — например `20260517-1430-choose-postgres.md`.

Дата+время убирают коллизии при работе в разных ветках. Сортировка естественная.

## Шаблон

При создании нового ADR — копируй [`template.md`](template.md):

```bash
TS=$(date +%Y%m%d-%H%M)
cp docs/adr/template.md "docs/adr/${TS}-my-slug.md"
```

Либо используй skill `/adr <title>` — создаст по шаблону, сразу заполнит дату и допишет строку в индекс `AGENTS.md`.

## Правила

- **Один ADR = одно решение.** Не пакуй несколько в один.
- **Пиши до или во время реализации**, не после. Иначе теряется контекст «почему».
- **Перезаписывая решение** — НЕ редактируй старый ADR. Создай новый, в старом меняй только `Status: Superseded by <new-adr>`.
- **`Alternatives considered`** — обязательно. Если ничего не рассматривали — решение слабое.
- **Секция `## Связки`** — обязательна. Ссылки на связанные документы.
